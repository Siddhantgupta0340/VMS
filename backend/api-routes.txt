AUTHENTICATION & TOKEN MATRIX
-----------------------------
All protected APIs require a JSON Web Token (JWT) in the Authorization header.
Header format:
  Authorization: Bearer <JWT_ACCESS_TOKEN>

ROLES & HIERARCHY
-----------------
* SUPER_ADMIN      : Complete system control, user CRUD, toggle status, bypass actions.
* CASE_MANAGER     : Creator role for Vendors, POs, Invoices, and Payments. Can only access own records.
* FINANCE_MANAGER  : Reviewer/Approver for Vendors, POs, Invoices, and Payments.
* L1 APPROVER      : Approves Invoices where Amount <= 10,000 INR.
* L2 APPROVER      : Approves Invoices where Amount <= 100,000 INR.
* L3 APPROVER      : Approves Invoices where Amount > 100,000 INR.

STANDARD ERROR RESPONSES
------------------------

1. Validation Error (Status: 400 Bad Request)
{
  "success": false,
  "status": 400,
  "message": "Validation failed: <Zod validation issues>"
}
Note: The validation middleware includes a safe fallback default `|| {}` for all wrapper objects (body, query, params) to avoid the generic "expected object, received undefined" type crash, ensuring specific validation issues are always returned in the JSON payload.

2. Unauthorized Error (Status: 401 Unauthorized)
{
  "success": false,
  "status": 401,
  "message": "Unauthorized"
}

3. Forbidden Error (Status: 403 Forbidden)
{
  "success": false,
  "status": 403,
  "message": "Forbidden"
}

4. Not Found Error (Status: 404 Not Found)
{
  "success": false,
  "status": 404,
  "message": "Resource not found"
}

5. Conflict Error (Status: 409 Conflict)
{
  "success": false,
  "status": 409,
  "message": "Resource already exists."
}

6. Internal Server Error (Status: 500 Internal Server Error)
{
  "success": false,
  "status": 500,
  "message": "Internal Server Error"
}

## USER LOGIN
---
* API Name: User Authentication / Login
* HTTP Method: POST
* Complete Endpoint: /api/v1/auth/login
* Description: Authenticate using email and password to receive JWT tokens.
* Bearer Token Required: NO
* Role Required: None (Public)
* Required Headers:
  Content-Type: application/json
* URL Parameters: None
* Query Parameters: None
* Complete Request Body:
{
  "email": "admin@vms.com",
  "password": "Admin@123"
}
  * Validation Rules:
    - email: Required, string, valid email format, trimmed, lowercased
    - password: Required, string, minimum length 8, complex character constraints
* Success Response (Status: 200 OK):
{
  "success": true,
  "message": "Login successful.",
  "data": {
    "user": {
      "id": "78326a27-0245-42f0-94eb-b6732be3ff81",
      "email": "admin@vms.com",
      "role": "SUPER_ADMIN",
      "first_name": "System",
      "last_name": "Admin",
      "is_active": true,
      "last_login_at": "2026-06-30T10:15:00.000Z",
      "created_at": "2026-06-30T10:10:00.000Z"
    },
    "accessToken": "eyJhbGciOiJIUzI1NiIsIn...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIsIn..."
  }
}
* Business Logic: Validate credentials. Hash check using bcrypt. Generate access/refresh tokens. Record last_login_at and refresh_token in database.
  - Controller Function: login()
  - Service Function: authService.login()
  - Repository Function: userRepository.findByEmail(), userRepository.updateUser()
  - Database Operation: findUnique, update
* cURL Request:
curl -X POST http://localhost:5000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "admin@vms.com", "password": "Admin@123"}'
* Postman Request: POST http://localhost:5000/api/v1/auth/login with raw JSON body
* Thunder Client Request: POST http://localhost:5000/api/v1/auth/login with raw JSON body
* Axios Example:
axios.post('http://localhost:5000/api/v1/auth/login', {
  email: 'admin@vms.com',
  password: 'Admin@123'
}).then(res => console.log(res.data));
* Fetch API Example:
fetch('http://localhost:5000/api/v1/auth/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email: 'admin@vms.com', password: 'Admin@123' })
}).then(res => res.json()).then(console.log);
* Expected Status Codes: 200 (Success), 400 (Bad request payload), 401 (Invalid credentials)

  - Repository Function: userRepository.findByEmail(), userRepository.updateUser()
  - Database Operation: findUnique, update

  -H "Content-Type: application/json" \

16. SECURITY REVIEW
================================================================================

1. Authentication
   - Secured via JWT Access Tokens with short expiration times.
   - Refresh tokens stored in DB for sliding-session invalidation.

2. Authorization
   - Controlled via role-based middleware (`authorize`).
   - Route path scoping limits data leakages (e.g. Case Managers can only access records created by them, L1/L2/L3 see only their assigned invoices).

3. Rate Limiting
   - Managed globally in Express. (Rate limiter setup exists in `src/app.js` via standard express-rate-limit).

4. Input Validation
   - Schema enforcement using Zod. Unsanitized properties are automatically stripped or throw 400 Bad Request before hitting controllers.

5. SQL Injection Protection
   - Handled via Prisma ORM parameterized queries. Native PostgreSQL statements are sanitized.

6. XSS / Security Headers
   - Helmet middleware is mounted globally to configure Content-Security-Policy (CSP), clickjacking, and mime-sniffing protection.

7. CSRF Protection
   - Not configured in current build. Add anti-CSRF token synchronization if frontend transitions from Bearer JWT in headers to HTTP-Only Cookie storage.

8. File Upload Security
   - Multer middleware is imported but unused in current endpoint list. Ensure MIME type validation is enforced on any future uploads folder implementation.
